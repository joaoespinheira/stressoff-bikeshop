## Commands

```bash
npm run dev          # Start dev server (Turbopack, default in Next.js 16)
npm run build        # Production build
npm run start        # Start production server
npm run lint         # ESLint
npm run db:generate  # Generate Prisma client after schema changes
npm run db:push      # Push schema to DB (dev, no migrations)
npm run db:migrate   # Create and apply migration (production)
npm run db:seed      # Seed DB with demo data (admin/chico/customer accounts)
npm run db:studio    # Open Prisma Studio GUI
```

## Architecture

### Next.js 16 Key Differences
- **`proxy.ts`** replaces `middleware.ts` for route interception (file renamed, function renamed `proxy`)
- **All Request APIs are async**: `await cookies()`, `await headers()`, `const { slug } = await params`
- **Turbopack** is the default bundler — no `--turbopack` flag needed
- **`next lint` is removed** — use `npm run lint` (ESLint CLI directly)
- **`refresh()`** from `next/cache` replaces `router.refresh()` in Server Actions
- **`cacheLife`/`cacheTag`** — no longer need `unstable_` prefix

### Auth Pattern
Custom JWT sessions via `jose` (no NextAuth). The session is a signed cookie.
- `lib/session.ts` — encrypt/decrypt JWT, create/delete session cookie
- `lib/dal.ts` — `verifySession()`, `verifyAdmin()`, `verifyMechanic()` — all use React `cache()`. Call these in Server Components and Server Actions to get the session.
- `app/actions/auth.ts` — `login`, `register`, `logout` Server Actions
- Route protection in `proxy.ts` (optimistic, cookie-based only — no DB calls)

### i18n
`next-intl` with `[locale]` folder. All pages live under `app/[locale]/`. Translation files in `messages/pt.json` and `messages/en.json`. The `i18n/request.ts` config resolves locale from the request. Use `useTranslations()` in Client Components and `getTranslations()` in Server Components.

### Database
Prisma v7 with PostgreSQL. The generated client outputs to the default `node_modules/@prisma/client`.
Always import `db` from `@/lib/db` (singleton pattern for dev).

### Dual-language content
Products, Categories, and Repair descriptions have `namePt`/`nameEn` and `descriptionPt`/`descriptionEn` fields. Always select the right field based on `locale`.

### Cart
Zustand store in `store/cartStore.ts` with `persist` middleware (localStorage). Import `useCartStore` in Client Components only. The `CartDrawer` is rendered in the locale layout and controlled by the store's `isOpen` state.

### Payments
Stripe Checkout Sessions (not Payment Intents directly). The flow is:
1. POST `/api/stripe/checkout` → creates a pending Order + Stripe session
2. User pays on Stripe-hosted page
3. Stripe sends webhook to `/api/stripe/webhook` → Order status updated to `PAID` + loyalty points awarded

### Human-readable IDs
Orders: `SB-YYYY-XXXXX`, Rentals: `SR-YYYY-XXXXX`, Repairs: `SP-YYYY-XXXXX`. Generated by `generateOrderNumber()` in `lib/utils.ts`.

### User Roles
`CUSTOMER` | `MECHANIC` | `ADMIN`. Role is embedded in the JWT. "Chico" is the head mechanic (`MECHANIC` role). Admin and mechanic can access `/admin/*`.

### PWA
`public/manifest.json` is static — no plugin needed. Icons should be placed in `public/icons/`. The app is installable as-is from the browser.

## Environment Variables
Copy `.env.local` and fill in real values. Required for full functionality:
- `DATABASE_URL` — PostgreSQL connection string
- `SESSION_SECRET` — random 32-char string (`openssl rand -base64 32`)
- `STRIPE_*` — from Stripe dashboard
- `CLOUDINARY_*` — from Cloudinary dashboard
- `NEXT_PUBLIC_WHATSAPP_NUMBER` — number without `+` (e.g. `351912345678`)
